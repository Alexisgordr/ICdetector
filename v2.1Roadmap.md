# ICdetection — v2.1 Ideas & Roadmap

## Status: v2.0 — Stable / Frozen

Version 2.0 is considered **stable and feature-complete** within the boundaries of what
Android userland allows without root access. The project is intentionally **frozen**: no
new heuristics or algorithmic changes will be added for now.

The v2.0 release added three statistical/historical hardening layers on top of the existing
engine, focused on improving the *quality* of detection and reducing false positives rather
than raising the detection ceiling (which is fixed by Android, not by the code):

- **Percentile-based baseline (H13):** anomalies must exceed the cell's own historical P99.
- **Cell reputation (trust score):** well-established clean cells dampen the noisy,
  instantaneous heuristics — never amplify, never touch physics-anchored heuristics.
- **RF fingerprint (RSRQ/SINR):** per-cell signal-quality signature; dormant until weeks of
  data accumulate, conservative by design.

---

## Plan: Data Collection Phase (2–3 months)

Before making any further changes, the project enters a deliberate **observation period of
2 to 3 months**. During this time the goal is **only**:

- Run v2.0 in real-world daily use.
- Collect field data (cell history, exported CSVs).
- Observe and study any false positives.
- Gather feedback from real usage.

**No new features or algorithmic changes during this phase.** The code will only be touched
for **bug fixes and small, targeted corrections** as issues are discovered in real-world use
(this is normal maintenance, not new development). Field testing has repeatedly proven more
valuable than synthetic tests for this project (it surfaced real issues such as the ARFCN /
carrier-aggregation false positive, which no unit test would have caught). The accumulated data
is what will tell us which future improvements are actually worth making — and how to calibrate
them.

After the collection phase, the exported CSV data will be reviewed to decide, **with real data
in hand**, which improvements to the analytical algorithms are justified.

## Future Improvement Ideas (only after the data phase, ordered by value / risk)

These are candidates for a future v2.1, **not** committed work. Each should be revisited only
once there is enough real-world data to justify and calibrate it. The guiding principle for any
future work: the remaining margin is **not** in finding a magic new signal to detect advanced
Stingrays — it is in getting better at distinguishing a *real anomaly* from *unusual but
legitimate behavior*. That is the hardest part of any anomaly detector.

### Tier 1 — Lowest risk, study first

1. **Evidence-quality score (separate from risk).**
   Distinguish *risk level* from *evidence quality*. A "high risk" verdict based on 2
   observations with a poor GPS fix and no history is very different from a "medium risk"
   verdict backed by 100 observations, solid history, and a good fix. This is purely
   informational — it adds interpretation context without changing any detection decision,
   so it is the safest place to start.

### Tier 2 — Valuable, but require the collected data to calibrate

2. **Bayesian weight recalibration from real usage.**
   The likelihood ratios are currently human-designed. With logged statistics (which
   heuristics fired, final score, how long the anomaly lasted, whether it recurred or
   normalized), the weights could be recalibrated from observed behavior — e.g. lower the
   weight of a heuristic that almost always normalizes on its own, raise the weight of
   combinations that tend to produce persistent events. No ML or architecture change needed;
   only observation and recalibration. **Requires the data phase first.**

3. **Smarter temporal decay of reputation.**
   Reputation currently uses a recent window. A more refined model would let trust decay over
   time when a cell disappears and later returns (networks change), e.g. `Trust(t) =
   Trust0 * exp(-t)` or a simplified variant, instead of inheriting full prior confidence.
   Low-to-medium risk; marginal refinement of the existing window-based forgetting.

### Tier 3 — Interesting but handle with care (probably later or never)

4. **Inter-heuristic coherence measure.**
   Explicitly measure whether heuristics reinforce each other (H11 + H15 + a sharp power
   change is far stronger than H11 alone). Note: the Bayesian engine already combines
   correlated evidence today, so this is a refinement of existing behavior, not a new signal.
   Medium risk (touches the engine).

5. **Per-operator dynamic profile.**
   Some operators behave differently (e.g. stable vs. frequently-changing PCI across sectors).
   Learning per-MCC/MNC thresholds could avoid penalizing one unfairly — but it adds state and
   complexity and risks false positives if mis-calibrated. The most likely to over-complicate
   the system; defer.

## Explicitly NOT planned

The following have been evaluated and rejected as poor value/risk for this project:

- **Handover / cell-transition graphs** — transition novelty is intrinsically false-positive
  prone (new routes are not attacks), and the robust part (geographic impossibility) is already
  covered by H11. Would need persisted transitions or noisy reconstruction.
- **More instantaneous RF heuristics, power rules, neighbor rules, latency rules** — the engine
  is at the point of diminishing returns; additional instantaneous rules add noise, not
  capability.
- **ARFCN/EARFCN as an identity signal** — proven unreliable in the field (carrier aggregation).
- **ProGuard/R8 minification** — pointless for a GPL open-source project; silent-breakage risk
  outweighs marginal APK savings.

---

## Guiding principles for any future work

- Never break working code; the v2.0 baseline has no false positives in normal real-world use.
- Additive and read-only by default; touch the DB schema only when truly necessary, and only
  with the established safe, non-destructive `ALTER TABLE ADD COLUMN` migration pattern.
- Conservative thresholds; when in doubt, err toward not firing.
- Validate every change against real exported CSV data, not just synthetic tests.
- The detection ceiling is set by Android (no RRC/NAS/baseband access), not by the code — future
  work improves evidence quality and false-positive control, not the ceiling.
