# ICdetection — v2.1 Roadmap

**Status:** v2.0 is frozen. No code changes are planned for the next 2–3 months except genuine bug fixes.

The goal of this frozen period is to collect field data, export CSVs, study false positives, and decide future changes with real evidence instead of guessing.

The remaining improvement margin is not a magic new signal against sophisticated IMSI-catchers. ICdetection is limited by Android userland: it cannot inspect RRC/NAS/baseband traffic like dedicated hardware tools can. The realistic goal for v2.1 is to improve internal coherence, reduce false positives, and make better use of local history.

---

## Already Shipped

These fixes are already part of v2.0 and should not be re-done:

- Database indexes on `(cid, mnc, tac, mcc)` and `(timestamp)` for faster history queries.
- `getRecords()` export wrapped in `try/catch` so malformed rows cannot crash history export.
- Coordinate backfill scoped only to the current observation.
- CSV field escaping according to RFC 4180.
- Version metadata aligned to v2.0: `versionCode = 2`, `versionName = "2.0"`.

## Tier 1 — Low Risk, High Value

### 1. Apply External Verification Before Analysis

`analyzeThreats()` currently runs on the freshly parsed cell, usually still marked as `PENDING`. The cached or database status from OpenCellID/WiGLE is applied afterwards for display.

This means a `VERIFIED` cell may not always receive its score bonus, and a `NOT_FOUND` cell may not always receive its small penalty during analysis.

**Goal:** resolve `knownStatus` before calling `analyzeThreats()` and pass the updated cell into the analyzer.

**Expected effect:** better internal consistency, not more aggressive detection.

---

### 2. Use Full Cell Identity Everywhere

Some history queries and cache keys still use `CID + MNC + TAC` without `MCC`. Other paths already use the full identity `CID + MNC + TAC + MCC`.

**Goal:** thread `MCC` through:

- `getPreviousCellHistory()`
- `getCellSignalBaseline()`
- `getCellReputation()`
- `getCellRfFingerprint()`
- related cache keys
- temporal confirmation keys

**Expected effect:** cleaner cell identity handling, especially for border, roaming, or rare operator edge cases.

---

### 3. History-Maturity Confidence Tiers

The app already has a trust/reputation concept. v2.1 can make it easier to interpret by exposing maturity tiers.

Possible tiers:

- `<7 days`: learning mode
- `7–30 days`: normal baseline
- `30–90 days`: strong baseline
- `90+ days`: very reliable baseline

**Goal:** avoid treating a cell seen 4 times like a cell seen hundreds of times across many days.

**Expected effect:** better user interpretation and more conservative trust handling.

---

### 4. Audit Log: Ciphering N/A

The UI already shows hardware ciphering as unavailable when Android/device APIs do not expose it. The terminal audit log can still show it as `FAILED`.

**Goal:** print `NO DISPONIBLE` when `hardwareCipheringAvailable == false`.

**Expected effect:** clearer reporting. No detection impact.

---

## Tier 2 — Valuable, Data-Dependent

### 5. Robust Baseline for H13

H13 currently uses mean/stddev plus percentiles. Mean/stddev can be affected by outliers.

**Goal:** evaluate median, MAD, and stronger percentile logic after enough field data is collected.

Optional idea: separate samples by approximate zone so the same cell seen from very different places does not create a noisy baseline.

**Expected effect:** fewer false positives from unusual but legitimate signal variation.

---

### 6. More Mature H15 PCI Logic

H15 currently focuses on PCI stability and ignores ARFCN because field data showed ARFCN can fluctuate legitimately due to carrier aggregation.

**Goal:** refine PCI logic to distinguish:

- one-time PCI change followed by stability
- recent PCI alternation
- new PCI plus abnormal power
- new PCI plus weak/absent neighbor environment

**Expected effect:** better detection of suspicious RF identity instability with fewer false positives.

---

### 7. Bayesian Weight Recalibration

The current likelihood ratios are reasoned estimates, not measured constants. After collecting real field data, they can be adjusted conservatively.

**Goal:** review which heuristics fire often, which normalize quickly, and which persist.

**Expected effect:** reduce overweighted noisy signals and preserve strong physical/contextual indicators.

---

### 8. Smarter Reputation Decay

A cell that was trusted months ago should not necessarily keep full confidence forever if it disappears and later returns.

**Goal:** add a simple time-decay model for trust.

Example:

- recent clean history = high trust
- old inactive history = gradually reduced trust
- returning cell = still remembered, but not blindly trusted

**Expected effect:** better long-term behavior without discarding useful history.

---

### 9. CSV Import / Restore

The app can export CSV, but not restore it. Import would turn export into a real backup system.

**Goal:** add CSV import only with strong safety rules:

- parse the same columns exported by `ExportUtils`
- support RFC 4180 escaped fields
- validate every row before insertion
- use a database transaction
- deduplicate rows
- offer a dry-run summary before writing
- reject malformed rows instead of guessing

**Expected effect:** safer migration between devices and recovery before experiments.

**Risk note:** this writes to the database, so it should be implemented carefully. It is additive, but not “zero risk”.

---

## Tier 3 — High Value, High Effort / Risk

### 10. Neighbor Fingerprint

A false base station may clone the serving cell, but it may not reproduce the surrounding RF ecosystem.

**Goal:** learn historical neighbor snapshots per cell/zone:

- neighbor PCI/ARFCN
- approximate relative powers
- number of neighbors
- repeated local RF patterns

Then flag major mismatches conservatively.

**Expected effect:** possible improvement against semi-professional setups.

**Risk:** high false-positive potential. Neighbor cells change legitimately often. This should only be attempted with enough real data.

---

### 11. Combination Scoring

The Bayesian scorer already combines evidence and groups correlated heuristics. A future layer could detect persistent multi-signal combinations.

**Goal:** recognize cases where several weak but independent anomalies persist together.

**Expected effect:** better handling of persistent multi-signal anomalies.

**Risk:** if naive, it can amplify false positives. This does not create a new signal against sophisticated catchers; it only improves interpretation when Android-visible traces already exist.

---

### 12. Transition Graph

Track normal transitions between cells, such as `A -> B -> C`, and flag unusual transitions only when combined with strong gating.

Possible gates:

- low speed
- strong signal
- repeated short-window transition
- suspicious neighbor context

**Expected effect:** maybe useful for local route anomalies.

**Risk:** very high false-positive potential. New routes, temporary coverage changes, and normal handovers can look unusual. Only consider after neighbor fingerprinting proves useful.

## Housekeeping

These items have no detection impact but may improve polish or maintainability:

- Update minor lint warnings.
- Update WiGLE `User-Agent` from `ICdetection/1.0` to current version.
- Add monochrome launcher icon.
- Consider moving user-facing strings to `strings.xml`.
- Consider instrumented/Robolectric tests for Android API paths.
- Consider splitting `MiniICService` only if future changes make it painful to maintain.
- Review/document the `WRITE_SECURE_SETTINGS` and `WRITE_SETTINGS` permissions. The former is not granted on a normal install (requires `adb`), so it is inert in practice, but both are powerful and may worry users/reviewers. Either document clearly why they exist (airplane-mode toggle) in the README, or remove them if the feature is not essential. Least-privilege hygiene, not a vulnerability.
- Consider storing the WiGLE API token in `EncryptedSharedPreferences` instead of plain `SharedPreferences`. Real-world risk is already low (MODE_PRIVATE + `allowBackup="false"` + non-debuggable release mean it cannot be extracted without root), so this is defense-in-depth, low priority.

## Not Planned

These ideas were considered and rejected for now:

- Using ARFCN as an identity signal in H15.
  - Field data showed ARFCN can fluctuate legitimately due to carrier aggregation.
- Adding more instant RF/power/neighbor rules without data.
  - Likely to increase false positives.
- Enabling ProGuard/R8 minification.
  - Low value for a GPL open-source project and possible silent breakage risk.
- Waking GPS on every cell change while the screen is off.
  - Would harm battery life and goes against the current low-drain design.
- Advertising or tracking monetization.
  - Against the privacy-first philosophy of the project.

---

## Long-Term Vision

A possible future direction is a privacy-preserving community anomaly database.

Users could voluntarily share anonymized/aggregated data to build known-legitimate tower baselines per city or region.

Hard requirements before this is even considered:

- no user movement traces
- no raw GPS history sharing
- aggregation to tower/area-level only
- anti-poisoning protections
- transparent opt-in design
- local-first behavior preserved

This is not a v2.1 feature. It would be a separate long-term project.
