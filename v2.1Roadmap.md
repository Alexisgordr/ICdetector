# ICdetection — v2.1 Roadmap

**Status:** v2.0 is frozen. No code changes are planned for the next 2–3 months except genuine bug fixes.

The goal of this frozen period is to collect field data, export CSVs, study false positives, and decide future changes with real evidence instead of guessing.

The remaining improvement margin is not a magic new signal against sophisticated IMSI-catchers. ICdetection is limited by Android userland: it cannot inspect RRC/NAS/baseband traffic like dedicated hardware tools can. The realistic goal for v2.1 is to improve internal coherence, reduce false positives, and make better use of local history.

---

## Already Shipped

These fixes are already part of v2.0 and should not be re-done:

- Database indexes on `(cid, mnc, tac, mcc)` and `(timestamp)` for faster history queries.
- `getRecords()` export wrapped in `try/catch` so malformed rows cannot crash history export.
- Coordinate backfill scoped only to the current observation.
- CSV field escaping according to RFC 4180.
- Version metadata aligned to v2.0: `versionCode = 2`, `versionName = "2.0"`.
- Ciphering/IMSI wording corrected: unavailable modem-security signals are shown as `N/A`, not as proof of failure or safety. The old dead callback methods (invented names that Android never invoked, so they detected nothing) were removed along with their misleading "(Reflexión)" logs, so the code no longer implies a capability it does not have. Ciphering reads `N/A` consistently in the UI and the audit terminal.
- **Persist alarm evidence to history (field-confirmed fix).** Root cause: history was only written on cell change (`cid != prevCid`), but threats confirm over 3 cycles on the *same* cell, so confirmed alarms left no trace in the CSV (the cell stayed logged as `100 / OK`). `checkAlerts()` now persists the triggering observation — real score + actual `suspiciousReason` — inside the `if (cell.isSuspicious)` block, so it fires regardless of handover. Logged once per episode (same `cellId`, reset on each handover) to avoid flooding the DB cycle-by-cycle. Persistence is gated to the temporally-confirmed path only (a `confirmed` flag on `checkAlerts()`): the post-API-verification re-analysis path still sounds its alert but does **not** write evidence, since its suspicion is raw single-cycle — so the forensic log contains 3-cycle-confirmed alarms only. Gating the write (rather than adding `applyTemporalConfidence()` to the second path) avoids double-incrementing the anomaly-streak counters. Alarms are now reviewable forensic evidence. *Pending real-world validation: confirm the next alarm exports a low-score row with its reason instead of `100 / OK`.*
- **Sanitize corrupt GPS fixes (field-confirmed fix).** A real fix landed ~1,100 km away (Pamplona observation carrying a Netherlands coordinate) — a spurious reading that can make a stationary cell appear to "teleport" and false-trigger H11 / ping-pong logic. `isPlausibleFix()` rejects physically implausible fixes (implied speed > 400 km/h vs the last accepted fix) on top of the existing accuracy/age filters, with self-healing after repeated rejections so a bad reference can't permanently block GPS. A rejected fix leaves the coordinate null ("unknown") instead of poisoning the geographic heuristics. The check now guards **all three GPS→DB paths**: the stream (`getCurrentLocation()`), the forced high-accuracy fix, and the on-GPS-available fill — so no corrupt coordinate enters by any route. Fails safe.
- **GPS-only location (root-cause fix for the recurring poison).** Field data showed the corrupt ~1,100 km Netherlands coordinate was not a one-off: the same fix (identical to 8 decimals) recurred across ~30 records and ~29 cells, always with weak signal — the signature of a cached/default position returned by the **network location provider** (GrapheneOS proxied network location) whenever GPS was unavailable. Network location is also *circular* for this app: it is partly derived from cell towers, the very thing the heuristics validate. The app now uses `GPS_PROVIDER` only — it no longer subscribes to `NETWORK_PROVIDER` in the location stream, and `getCurrentLocation()` no longer reads `getLastKnownLocation(NETWORK_PROVIDER)`. With no valid GPS fix the coordinate is left null ("unknown") and fails safe. Trade-off: more null coordinates indoors / under weak GPS, which is correct for a forensic tool (no data > wrong data); SUPL/PSDS keep GPS time-to-first-fix fast. This removes the poison source in code, independent of device location settings. *Note: existing poisoned rows already in the DB are not retroactively cleaned — starting collection from a fresh/cleared history is recommended so the multi-month baseline is uncontaminated.*
- **Persist the GPS plausibility reference across restarts (field-confirmed fix).** `isPlausibleFix()` validates a fix by implied speed vs the last accepted fix, but it accepts the first fix unconditionally when it has no reference (`lastAcceptedLocation == null`) — you cannot compute a speed from a single point. That field was in-memory only, so every service restart reopened the window: right after a restart a stale/cached corrupt fix (the same Netherlands coordinate) could be accepted as the *first* fix and written to the DB, with no reference to reject it against. This was the route by which a bad coordinate still entered despite the sanitization above. The accepted reference is now persisted to `prefs` (`lat,lon,time`, written only when the fix is newer than the last persisted one, to avoid redundant writes from the repeated `getLastKnownLocation`) and reloaded in `onCreate()`, so the first post-restart fix is speed-checked instead of trusted blindly. A staleness guard ignores a persisted reference older than 6 h (or future-dated) so legitimate long travel with the app closed is not falsely rejected. Both acceptance sites (`getCurrentLocation()`, the forced high-accuracy fix) route through a single `acceptLocation()`. No schema/DB change. Self-healing is preserved: the 3-strike auto-recovery and the 6 h guard release a poisoned reference if one ever persists. *Note: a full history reset also clears `prefs`, so that one restart is bootstrap again; a normal restart keeps the reference.*
- **GPS-to-cell race fix.** `onGpsAvailable()` captured the active cell *before* a 3 s GPS-stabilization delay and then wrote coordinates using it. If a handover happened during the wait, a fresh coordinate (and the re-verification) was attributed to the *previous* cell, injecting a real position into the wrong antenna's records (noise for H11/H13). The active cell is now re-captured *after* the delay, so coordinates and verification always belong to the actually-active cell.
- **Full cell identity in history queries and prefetch cache (was Tier 1 #1 — it could not wait).** Four historical queries filtered on `CID + MNC + TAC` without `MCC` — `getPreviousCellHistory()`, `getCellSignalBaseline()`, `getCellReputation()`, `getCellRfFingerprint()` — unlike `getCellRfStability()` and `getKnownStatus()`, which already used the full identity. Since `MNC` is only unique within an `MCC`, this could merge distinct cells in border / roaming / shared-network / multi-SIM cases, contaminating H11/H13, reputation and the RSRQ/SINR fingerprint. `MCC` is now threaded through all four (WHERE clause + signatures + all six call sites), and the in-memory prefetch cache signature (`cachedCellSignature`) now includes `MCC` to match `rfSig`, so history/baseline are not reused across a different `MCC` within the cache TTL at the same location. No schema change: the `mcc` column and the `(cid,mnc,tac,mcc)` index already existed, so `DATABASE_VERSION` is unchanged and no migration was added. *Recommended: start collection from a fresh/cleared history so pre-fix rows with `mcc='N/A'` do not sit orphaned relative to real-MCC queries. The temporal-confirmation streak keys are the one remaining piece — see Tier 1 #1.*
- **Latency indicator honesty + VPN/Tor gating.** The "RED OK" indicator was bound only to the latency-probe state, which defaulted to `OK` and was never updated while the probe was disabled — so with latency off the UI showed a green "RED OK" even though nothing verified the network. That is a false assurance in a forensic tool, and inconsistent with how ciphering is shown as `N/A` rather than assumed safe. The latency probe is also not representative when traffic does not egress straight to the cellular network. Fixes: the network state initializes to `N/A`; `checkLatencyAnomaly()` now skips measuring and reports `N/A` whenever the probe cannot measure reliably — disabled, **or WiFi / VPN / Tor (SOCKS5) active** (previously WiFi only) — with the guard running before the 30 s throttle so the indicator updates immediately; the UI shows three states (`RED OK` green / `RED ANÓMALA` red / `RED N/A` neutral grey) and the null-service fallback is `N/A`; handover resets no longer force a false `OK`; and the Settings "Detección de Latencia" toggle is greyed out and non-clickable while a VPN is active, mirroring the existing WiFi handling. No detection logic changed: heuristic 12 and the RF warning read `== "ANOMALA"`, so `N/A` counts as non-anomalous.
- **Telephony callback watchdog fixed.** The watchdog used a boolean (`isCallbackWorking`) that latched `true` after the first event and never reset, so it never re-fired. If the radio HAL dropped the telephony callback mid-session (common after Doze or in multi-day runs), it was never re-registered and **data collection stopped silently**. Replaced with a real liveness timestamp: if no callback arrives for 30 s, callbacks are unregistered and re-registered. The periodic service loop drives the check independently of the callbacks, so it runs even when the callback is dead. (This is the *telephony* watchdog — distinct from the future cipher callbacks in Tier 1 #3.)
- **H8 / H3 false-positive fixes (field-aligned).** H8 no longer penalizes LTE EARFCN 0 — it is a valid Band 1 channel (2110 MHz); the modem's "unavailable" value arrives as a large number and is still rejected. 5G NR-ARFCN 0 is still rejected. H3 no longer flags MCC inconsistency when the serving cell's MCC has not been parsed yet (`"N/A"`), which was producing spurious `-30` hits during transient modem read delays. `BandPlan.earfcnToBandLte()` now maps EARFCN 0 to Band 1 for coherence, and the test suite was updated to defend the new behavior (LTE 0 passes, 5G 0 fails, EARFCN 0 → Band 1).
- **Active cell protected from cache eviction.** `pruneCaches()` evicts arbitrary entries from two unordered `ConcurrentHashMap`s; on a long trip (>`MAX_TRACKED_CELLS` distinct cells) it could drop the currently-camped cell's latency baseline / verification status, forcing a needless re-verification. It now excludes the active cell from eviction, using each cache's real key — bare `cellId` for `latencyHistory`, composite `mcc-mnc-tac-cellId` for `verificationCache`. Cap behavior unchanged.
- **Crash hardening (invisible, no behavior change).** `startForeground()` wrapped in `try/catch` (Android 14+ can throw if the location permission is missing or the OS restarts the service restricted) — on failure it stops cleanly via `stopSelf()` instead of crashing. Added a `CoroutineExceptionHandler` to the service scope so an uncaught coroutine exception is logged, not fatal. `MainActivity.startAndBindService()` guarded against double-bind (an `isBinding` flag) and against a throwing/failing `startForegroundService()`/`bindService()`, stopping the service if binding fails so it is not left orphaned.
- **Notification repaint throttle.** `updateNotificationText()` now repaints at most once per 2 s. The telephony callback can fire many times per second in transition zones; unthrottled repaints flooded the system server (IPC + battery) and tripped Android's "notification rate limit exceeded". DB logging, audit logs and the alarm **tone** stay real-time — only the visual repaint is throttled.
- **Apply known verification status before threat analysis (Tier 1 #1, shipped).** `analyzeThreats()` ran on the freshly parsed cell (still `PENDING`), and the cached `VERIFIED`/`NOT_FOUND` status was only applied afterwards for display. So the recorded `securityScore` did not include the verification bonus/penalty: a `VERIFIED` cell could miss up to **+15** (and, when very close to the tower with strong signal, also keep a `-15` "Proximidad anómala TA" that should be suppressed), occasionally flipping a legitimate verified cell below the 70 suspicious threshold — a false positive in the data. In the field this was masked because cells mostly score 100 (capped), but it could bite in weak-signal areas. The active cell is now enriched with its cached known status **before** analysis, so the score reflects verification within the same cycle. New cells remain `PENDING` (genuinely unverified) — that is correct. *Note: the original roadmap framed this as "internal consistency, not more aggressive detection"; on closer inspection it actually **reduces false positives**, so its value was slightly higher than first stated.*

## Tier 1 — Low Risk, High Value

### 1. Full Identity in Temporal-Confirmation Keys (residual of the MCC work)

The history queries and the prefetch cache signature now use the full `CID + MNC + TAC + MCC` identity (shipped — see Already Shipped). The one remaining spot is the temporal-confirmation streak map (`anomalyStreaks` / `lastStreakCellId` in `applyTemporalConfidence()`), which still keys on bare `cellId` and resets only when `cellId` changes.

**Goal:** key the streak map (and its reset condition) on the full identity, so a same-`CID` transition across a different `MNC` / `TAC` / `MCC` (border / operator edge) does not carry a streak over.

**Expected effect:** correctness in rare cross-operator edge cases.

**Impact:** low in practice — the streak resets on every `cellId` change and only one cell is camped at a time, so contamination needs the same `CID` reused under a different operator without a `cellId` change. Cheap to close for completeness.

---

### 2. History-Maturity Confidence Tiers

The app already has a trust/reputation concept. v2.1 can make it easier to interpret by exposing maturity tiers.

Possible tiers:

- `<7 days`: learning mode
- `7–30 days`: normal baseline
- `30–90 days`: strong baseline
- `90+ days`: very reliable baseline

**Goal:** avoid treating a cell seen 4 times like a cell seen hundreds of times across many days.

**Expected effect:** better user interpretation and more conservative trust handling.

---

### 3. Advanced Modem Security Callbacks

The UI and audit log now treat hardware ciphering as `N/A` when Android/device APIs do not expose the signal. ICdetection v2.0 does not claim direct null-cipher or IMSI-disclosure detection on standard no-root Android installs.

A placeholder callback path is kept for future Android 16/API 36 modem-security APIs, but it does not currently generate alerts or affect scoring. (Note: this is unrelated to the telephony-callback watchdog already fixed in "Already Shipped".)

**Goal:** implement real `SecurityAlgorithmsListener` / `CellularIdentifierDisclosedListener` support only when Android 16+ hardware and permissions can be tested properly. Note these APIs likely require `READ_PRIVILEGED_PHONE_STATE`, which a sideloaded app cannot hold — so this may remain infeasible on normal installs, and Android 16 already rejects null ciphers at the OS level.

**Expected effect:** future direct modem-security visibility where the OS and device actually expose it. No current v2.0 detection impact.

## Tier 2 — Valuable, Data-Dependent

### 5. Robust Baseline for H13

H13 currently uses mean/stddev plus percentiles. Mean/stddev can be affected by outliers.

**Goal:** evaluate median, MAD, and stronger percentile logic after enough field data is collected.

Optional idea: separate samples by approximate zone so the same cell seen from very different places does not create a noisy baseline.

**Expected effect:** fewer false positives from unusual but legitimate signal variation.

---

### 6. More Mature H15 PCI Logic

H15 currently focuses on PCI stability and ignores ARFCN because field data showed ARFCN can fluctuate legitimately due to carrier aggregation.

**Goal:** refine PCI logic to distinguish:

- one-time PCI change followed by stability
- recent PCI alternation
- new PCI plus abnormal power
- new PCI plus weak/absent neighbor environment

**Expected effect:** better detection of suspicious RF identity instability with fewer false positives.

---

### 7. Bayesian Weight Recalibration

The current likelihood ratios are reasoned estimates, not measured constants. After collecting real field data, they can be adjusted conservatively.

**Goal:** review which heuristics fire often, which normalize quickly, and which persist.

**Expected effect:** reduce overweighted noisy signals and preserve strong physical/contextual indicators.

---

### 8. Smarter Reputation Decay

A cell that was trusted months ago should not necessarily keep full confidence forever if it disappears and later returns.

**Goal:** add a simple time-decay model for trust.

Example:

- recent clean history = high trust
- old inactive history = gradually reduced trust
- returning cell = still remembered, but not blindly trusted

**Expected effect:** better long-term behavior without discarding useful history.

---

### 9. CSV Import / Restore

The app can export CSV, but not restore it. Import would turn export into a real backup system.

**Goal:** add CSV import only with strong safety rules:

- parse the same columns exported by `ExportUtils`
- support RFC 4180 escaped fields
- validate every row before insertion
- use a database transaction
- deduplicate rows
- offer a dry-run summary before writing
- reject malformed rows instead of guessing

**Expected effect:** safer migration between devices and recovery before experiments.

**Risk note:** this writes to the database, so it should be implemented carefully. It is additive, but not “zero risk”.

---

## Tier 3 — High Value, High Effort / Risk

### 10. Neighbor Fingerprint

A false base station may clone the serving cell, but it may not reproduce the surrounding RF ecosystem.

**Goal:** learn historical neighbor snapshots per cell/zone:

- neighbor PCI/ARFCN
- approximate relative powers
- number of neighbors
- repeated local RF patterns

Then flag major mismatches conservatively.

**Expected effect:** possible improvement against semi-professional setups.

**Risk:** high false-positive potential. Neighbor cells change legitimately often. This should only be attempted with enough real data.

---

### 11. Combination Scoring

The Bayesian scorer already combines evidence and groups correlated heuristics. A future layer could detect persistent multi-signal combinations.

**Goal:** recognize cases where several weak but independent anomalies persist together.

**Expected effect:** better handling of persistent multi-signal anomalies.

**Risk:** if naive, it can amplify false positives. This does not create a new signal against sophisticated catchers; it only improves interpretation when Android-visible traces already exist.

---

### 12. Transition Graph

Track normal transitions between cells, such as `A -> B -> C`, and flag unusual transitions only when combined with strong gating.

Possible gates:

- low speed
- strong signal
- repeated short-window transition
- suspicious neighbor context

**Expected effect:** maybe useful for local route anomalies.

**Risk:** very high false-positive potential. New routes, temporary coverage changes, and normal handovers can look unusual. Only consider after neighbor fingerprinting proves useful.

---

### 13. Forensic Capture During Ping-Pong (without relaxing confirmation)

A rapid ping-pong between a suspicious cell and a legitimate one resets the per-cell streak on each bounce, so the suspicious cell never reaches 3-cycle confirmation and no forensic row is written (the ping-pong itself is still alerted separately). Relaxing the confirmation (e.g. accumulating streaks across bounces) would catch it but **increases false positives** for legitimate edge cells that occasionally dip — against the project's false-positive-control philosophy.

**Goal:** capture forensic evidence of the suspicious observations during a ping-pong event *linked to the existing ping-pong alert*, **without** changing the 3-cycle confirmation logic.

**Risk:** must be designed carefully so it logs evidence without re-introducing unconfirmed noise into the forensic history. Only with real data.

## Housekeeping

These items have no detection impact but may improve polish or maintainability:

- Update minor lint warnings. (Several were already cleared: the redundant `inner` modifier and the now-callerless `triggerSecurityAlert`, kept `@Suppress("unused")` as the reserved v2.1 cipher-alert path.)
- Update WiGLE `User-Agent` from `ICdetection/1.0` to current version.
- Add monochrome launcher icon.
- Consider moving user-facing strings to `strings.xml`.
- Consider instrumented/Robolectric tests for Android API paths.
- Consider splitting `MiniICService` only if future changes make it painful to maintain.
- Review/document the `WRITE_SECURE_SETTINGS` permission. It is not granted on a normal install and requires `adb`, so it is inert in practice. It exists only for the optional airplane-mode fallback, but because it is a powerful privileged permission, it should be clearly documented or removed if the feature is not essential. Least-privilege hygiene, not a vulnerability.
- Consider storing the WiGLE API token in `EncryptedSharedPreferences` instead of plain `SharedPreferences`. Real-world risk is already low (MODE_PRIVATE + `allowBackup="false"` + non-debuggable release mean it cannot be extracted without root), so this is defense-in-depth, low priority.
- Consider hoisting the Timing-Advance regex in `CellParser` to a compiled constant, reusing a single `FloatArray(1)` in distance loops, and moving date formatting to a shared thread-safe `DateTimeFormatter`. Micro-optimizations only (the service cost is dominated by radio/GPS, not these); no correctness impact. Optionally migrate the remaining `try/finally` cursors to `.use {}` for consistency (current pattern does not leak).

## Not Planned

These ideas were considered and rejected for now:

- Using ARFCN as an identity signal in H15.
  - Field data showed ARFCN can fluctuate legitimately due to carrier aggregation.
- Adding more instant RF/power/neighbor rules without data.
  - Likely to increase false positives.
- Enabling ProGuard/R8 minification.
  - Low value for a GPL open-source project and possible silent breakage risk.
- Waking GPS on every cell change while the screen is off.
  - Would harm battery life and goes against the current low-drain design.
- Holding a permanent `PARTIAL_WAKE_LOCK` to keep the CPU awake 24/7.
  - Would destroy battery life and make multi-month collection unusable; contradicts the deliberate low-drain design. Screen-off sampling is intentionally sparser as a battery trade-off, and network activity (the relevant moments) wakes the CPU anyway.
- Advertising or tracking monetization.
  - Against the privacy-first philosophy of the project.

## Long-Term Vision

A possible future direction is a privacy-preserving community anomaly database.

Users could voluntarily share anonymized/aggregated data to build known-legitimate tower baselines per city or region.

Hard requirements before this is even considered:

- no user movement traces
- no raw GPS history sharing
- aggregation to tower/area-level only
- anti-poisoning protections
- transparent opt-in design
- local-first behavior preserved

This is not a v2.1 feature. It would be a separate long-term project.
