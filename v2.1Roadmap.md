# ICdetection — v2.1 Roadmap

**Status:** v2.0 is frozen. From here, no code changes for 2–3 months except genuine bug fixes.
The goal of that period is to collect field data (CSVs) and study false positives. Every item below
is to be decided **with real data in hand**, not applied blind. Ordered by value / risk.

Guiding rule for all of it: the remaining margin is **not** a magic new signal against sophisticated
catchers — it is getting better at telling a *real anomaly* from *unusual but legitimate behavior*,
and improving internal coherence. The detection ceiling is set by Android (no RRC/NAS/baseband),
not by the code.

---

## Already shipped (not pending — listed so they aren't re-done)

- DB indexes on (cid, mnc, tac, mcc) and (timestamp) — query performance for a growing table.
- getRecords() export wrapped in try/catch — a malformed row can't crash the export.
- Coordinate backfill scoped to the current observation only (fills the latest row of the active
  cell if it lacks lat/lon; never mass-rewrites history or fills older rows).
- CSV field escaping (RFC 4180 csvEscape) — no field can misalign a row.
- Version metadata aligned to 2.0 (versionCode 2, versionName "2.0").

## Tier 1 — Low risk, high value (do first in v2.1)

### 1. Apply external verification (knownStatus) before analyzeThreats
Currently analyzeThreats runs on the freshly parsed cell (usually PENDING), and the OpenCellID/WiGLE
status (VERIFIED / NOT_FOUND) is resolved afterwards — so a VERIFIED cell may not always get its
bonus and a NOT_FOUND its small penalty in the score. Not detection-breaking (the strong heuristics
are the physics/history-anchored ones, not the optional API check), but a real internal-coherence
gap. Fix: resolve knownStatus first, then pass it into analyzeThreats. Improves consistency, not
aggressiveness. Top candidate.

### 2. Full cell identity (MCC) in all history queries
getPreviousCellHistory(), getCellSignalBaseline(), getCellReputation() and getCellRfFingerprint()
(plus their caches, and the temporal-confirmation keyed on cellId) use CID+MNC+TAC without MCC.
updateNullCoordinates() already includes MCC. Low real-world impact (same CID+MNC+TAC across
different MCC is rare — only border/roaming), and partly absorbed by the 3-cycle confirmation. Fix:
thread mcc through those functions, call sites and cache keys so every identity lookup is
CID+MNC+TAC+MCC. Clean and logical; not urgent.

### 3. CSV import / restore (turns export into a real backup)
The app exports CSV but cannot import. Adding an import path makes the existing export a restorable
backup: export before any risky change and re-import to recover; export on the old phone, import on
the new one. Closes the data-loss worry. Implementation: parse the exported columns (keep in sync
with ExportUtils, incl. RSRQ/SINR), re-insert via logConnection-style writes inside a single
transaction, with validation/dedup so a malformed/duplicate row can't corrupt the DB. Pairs with the
csvEscape already shipped, so round-tripping is lossless. Low risk if additive and validated.

### 4. History-maturity confidence tiers
Extend the existing trustFactor: <7 days = learning mode, 7–30 = normal, 30–90 = strong baseline,
90+ = very reliable. So the app doesn't judge a cell seen 4 times the same as one seen 500 times
over 40 days. Natural extension of the reputation system; low risk.

## Tier 2 — Valuable, but require the collected data to calibrate

### 5. Robust baseline (median / MAD / percentiles, zone-separated)
Replace mean/stddev in H13 with median + MAD (median absolute deviation), more resistant to
outliers, building on the percentile work already shipped. Optionally separate samples by approximate
zone so a cell seen far (weak) and near (strong) doesn't blur into one noisy baseline. Data-dependent
calibration.

### 6. More mature H15 (PCI)
Refine getCellRfStability + H15 to distinguish: one-time PCI change then stable (likely legit
reconfig) vs recent alternation (suspicious) vs new PCI with abnormal power, or with absent
neighbors (more suspicious). Medium risk; needs data.

### 7. Bayesian weight recalibration from real usage
Log which heuristics fire, the final score, how long anomalies last, whether they recur or normalize;
then recalibrate the likelihood ratios from observed behavior (lower the weight of heuristics that
almost always self-normalize, raise combinations that tend to persist). No ML, no architecture
change — observation + recalibration. Requires the data phase first.

### 8. Smarter temporal decay of reputation
Let trust decay when a cell disappears for a long time and later returns, instead of inheriting full
prior confidence (e.g. Trust(t) = Trust0 * exp(-t) or a simplified variant). Low-to-medium risk;
refines the existing recent-window forgetting.

---

## Tier 3 — High value but high effort / risk (study carefully, only with data)

### 9. Neighbor fingerprint
Learn the historical RF environment per cell/zone (which neighbors, approximate PCI/ARFCN, relative
powers) and flag mismatches. A catcher can clone the serving cell but not the whole surrounding
ecosystem — strongest proposed gain vs semi-pros. Needs a new table/snapshot and a new heuristic;
must be very false-positive-careful. Significant work, not a patch.

### 10. Combination scoring ("strong combos")
Boost probability when 3+ independent weak signals co-occur, beyond the current grouped-LR sum.
Conceptually strong vs sophisticated, but risky to tune (can amplify false positives if naive). The
Bayesian already combines evidence; this is an explicit co-occurrence layer on top.

---

## Reconsidered, still cautious

### 11. Transition graph (A→B→C routes)
Flag an unseen transition combined with low speed + strong signal. Previously rejected as
false-positive-prone (new routes are not attacks); the speed+signal gating makes it more defensible,
but it remains the riskiest item and needs a lot of data. Only after neighbor-fingerprint proves the
snapshot approach works.

## Cosmetic / housekeeping (whenever, zero detection impact)

- Audit-log shows ciphering as FAILED when it is N/A: print "NO DISPONIBLE" when
  hardwareCipheringAvailable == false (UI already handles this; only the terminal log doesn't).
- Lint warnings: core-ktx update available, obsolete values-v24/v26 folders, monochrome icon hint,
  KTX suggestion in MapViewer. All minor; clean up in a maintenance pass.
- Optional: split MiniICService (~1.5k LOC) into smaller components (telemetry, latency, GPS, alarms)
  for maintainability/testability. Big refactor; only if the project keeps growing.
- Optional: i18n (English/multi-language UI via strings.xml) to widen reach. Pure UI, no logic risk.
- Optional: instrumented tests (Robolectric) for the parser and the telephony callbacks, to cover
  the Android-API paths the 73 JVM tests can't.

---

## Not planned (evaluated and rejected)

- ARFCN as an identity signal (field data: carrier aggregation makes it unreliable — H15 uses PCI only).
- More instantaneous RF / power / neighbor / latency rules (diminishing returns; adds noise).
- ProGuard/R8 minification (pointless for a GPL open-source project; silent-breakage risk).
- Waking GPS on every cell change while the screen is off (reverses a deliberate battery
  optimization; uncertain benefit under Doze; real battery cost while moving in pocket — and in
  repose you're stationary so no coordinates are actually lost that matter).

---

## Long-term vision (v3.0 or a sibling project)

Community crowdsourced anomaly database: users share anonymized CSVs to build a large per-city
baseline of known-legitimate towers, downloadable so the app works with context from day one.
**Hard prerequisite before anything else here: privacy.** CSVs contain the user's GPS positions;
sharing must anonymize/aggregate to tower-level info only, never the user's location trace — for a
privacy app this is non-negotiable. Also needs anti-poisoning (multi-user consensus) and server
infrastructure. A project in itself; do it in phases, privacy first.
